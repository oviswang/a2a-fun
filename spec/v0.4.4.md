# a2a.fun Spec v0.4.4 (v0.4.3 + exchange framework fields only) — Draft

v0.4.4 is **behavior-identical** to v0.4.3.

**Hard requirement:** v0.4.4 introduces **fields only** (an “exchange framework” extension point). It MUST NOT define or trigger any behavior.

**Hard constraint (normative):**
- Exchange fields **MUST NOT influence state machine behavior**.
- Exchange fields are **informational only**.

Implementations:
- MAY include these fields in messages.
- MUST ignore these fields for decision-making.
- MUST NOT auto-start any exchange workflow.

---

## 1) Envelope additions (fields only)

In v0.4.4, the envelope MAY include an optional `x` (extensions) object.

```json
{
  "v": "0.4.4",
  "type": "...",
  "msg_id": "...",
  "session_id": "...",
  "ts": "...",
  "from": {"actor_id": "...", "key_fpr": "..."},
  "to": {"actor_id": "...", "key_fpr": "..."},
  "crypto": {"enc": "aead", "nonce": "..."},
  "body": {"ciphertext": "..."},

  "x": {
    "exchange": {
      "framework_v": "0",
      "cap": {
        "supports_exchange": false,
        "kinds": []
      },
      "hint": {
        "preferred_kind": null,
        "ui_text": null
      }
    }
  },

  "sig": "..."
}
```

### Field semantics (NO behavior)
- `x.exchange.framework_v`:
  - string version for the exchange extension field layout.
- `x.exchange.cap.supports_exchange`:
  - boolean capability flag only.
- `x.exchange.cap.kinds`:
  - list of strings describing potential future exchange kinds (e.g. `"profile"`, `"files"`, `"calendar"`).
  - MUST be treated as informational.
- `x.exchange.hint.preferred_kind`:
  - a hint string for UI or future negotiation; MUST NOT trigger actions.
- `x.exchange.hint.ui_text`:
  - optional text for UI display; MUST be non-identifying; MUST NOT contain raw handles.

---

## 2) Body additions (fields only)

Within decrypted message bodies, types MAY also include an optional `exchange` field.

Example (in `probe.hello` decrypted body):
```json
{
  "protocols": ["a2a.friendship/1"],
  "transports": ["webrtc"],

  "exchange": {
    "supports": false,
    "kinds": [],
    "note": null
  }
}
```

### Field semantics (NO behavior)
- These fields are purely declarative and MUST NOT:
  - request data
  - initiate any workflow
  - change state machine transitions
  - bypass human entry gating

---

## 3) Raw handle prohibition still applies
All v0.4.3 prohibitions remain.
- Exchange fields MUST NOT carry raw handles.
- Exchange fields MUST NOT carry contact methods.

---

## 4) Exchange field whitelist (decrypted body)

To avoid accidental behavior coupling, v0.4.4 defines a strict whitelist for which **message bodies** may include an `exchange` object.

### Allowed (whitelist)
- `probe.hello`
- `probe.summary`

### Disallowed (MUST NOT include `exchange` in decrypted body)
- `probe.question`
- `probe.answer`
- `probe.done`
- `human.entry`
- `human.exit`
- `friendship.establish`
- `session.close`
- `error`

> Rationale: keep exchange capability hints confined to probe metadata only, and prevent any temptation to influence consent, closure, or friendship creation.

## 5) Compatibility
- A v0.4.3 node receiving v0.4.4 messages should be able to ignore unknown fields.
- A v0.4.4 node MUST remain compatible with v0.4.3 behavior.
