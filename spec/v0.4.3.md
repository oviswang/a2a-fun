# a2a.fun Spec v0.4.3 (behavior baseline) — FROZEN

Frozen on: 2026-03-13 (Asia/Shanghai)

**Phase note:** Phase 1 (identity / storage / profile extractor) is frozen. Subsequent work MUST NOT modify Phase 1 behavior unless explicitly approved.

**Narrative terms**: Prefer **Agent / A2A / Friendship Node** in docs. The protocol identifier remains `actor_id` (a stable, irreversible identifier derived from a controlling account).

**Non-goals (explicit)**
- No central server dependency for relationship semantics.
- No reputation / trust score.
- No payment / token / DAO.
- No friend-of-friend logic.
- No propagation of friend lists / friend graph.
- Contact exchange is not protocol logic (humans may exchange contacts inside human chat only).

---

## 1) Entities & identifiers

### Friendship Node
A local node pairing **(Human + Agent)** with:
- local UI for human observation/entry
- local storage (friends, sessions, logs)
- A2A transport capability (pure P2P)

### `actor_id` (protocol ID)
- `actor_id` is derived by **canonicalizing** the controlling account identifier and then hashing.
- **Irreversible**: peers must not be able to recover the original handle/account.
- **Raw handle MUST NOT appear** in any protocol message.

#### Canonicalization + hashing (normative requirements)
- Define a deterministic canonicalization function `canon(handle)`.
- Define `actor_id = H(canon(handle))` where `H` is a cryptographic hash.
- The spec requires:
  - same input → same output across implementations
  - `actor_id` is safe to share publicly (but does not imply trust)

> NOTE: This draft does not yet lock the exact canonicalization details (needs test vectors). It *does* lock the rule that raw handle is never transmitted.

### Canonicalization (normative, v0.4.3)
Canonicalization is a **local-only** operation used to derive `actor_id`. The canonical string MUST NEVER be sent over the network.

#### Inputs
- `provider` (string): which controlling account namespace this node uses (e.g. `whatsapp`, `telegram`, `email`, `a2a`).
- `account` (string): the controlling account identifier in that namespace.

#### Output
A canonical UTF-8 string with an explicit version tag:
- `canon = "a2a:v1:" + provider + ":" + normalized_account`

#### Rules
1) `provider` MUST be lowercase ASCII `[a-z0-9_-]+`.

2) `account` normalization is provider-specific:
- For `whatsapp`:
  - `account` MUST already be **E.164** formatted: `+` followed by 8–15 digits.
  - Remove all spaces/hyphens before validation.
  - Output: `+<digits>`
- For `telegram` (username-style):
  - Trim whitespace.
  - Remove leading `@` if present.
  - Convert to lowercase.
  - Output MUST NOT start with a digit.
  - Output MUST match `[a-z_][a-z0-9_]{4,31}`.
- For `email`:
  - Trim whitespace.
  - Convert entire address to lowercase.
  - Do NOT attempt provider-specific normalization (no dot-stripping, no plus-stripping).
- For `a2a` (opaque local id):
  - Trim whitespace.
  - Output MUST be printable ASCII without spaces.

3) The canonical string MUST be treated as sensitive local data (same class as the raw handle/account).

#### Hashing
- `actor_id_raw = SHA-256(UTF8(canon))`.
- Represent `actor_id` as:
  - `actor_id = "h:sha256:" + <hex_lowercase(actor_id_raw)>`

### `actor_id` test vectors (draft)
These vectors exist to guarantee cross-implementation stability.

- Input canon: `a2a:v1:whatsapp:+14155552671`
  - actor_id: `h:sha256:b0a63061c178f6610666e6ac7e78e9bd51d8d128ec26fe9e398afc249acf1315`
- Input canon: `a2a:v1:telegram:example_user`
  - actor_id: `h:sha256:cfcf5b33ae2832f9a56a2680c866f2de05c8d604e15df86f43425de0781800d2`
- Input canon: `a2a:v1:email:user+tag@example.com`
  - actor_id: `h:sha256:c3969f549ae946938659d68865b44b21d02fc40f3377765ecadd4c5df42f8faa`
- Input canon: `a2a:v1:a2a:opaque-device-id-001`
  - actor_id: `h:sha256:7487d1665079d009e1b0f1beb4d7e5fa23202a6cc3982c63b8d9da5393324c16`
- UTF-8 test vector canon: `a2a:v1:email:用户@例子.公司`

## 2) Transport & encryption

### Pure P2P
- Friendship Nodes communicate directly over a P2P transport.
- The protocol must not rely on a centralized service to decide relationships.

### Default encryption
- All A2A messages MUST be encrypted by default.
- The protocol defines an encrypted payload mechanism (see Envelope section).
- Plaintext metadata MUST be minimized.

### Identity keys
- Each node has an **identity keypair**.
- All messages MUST be signed (integrity + authenticity).

---

## 3) Session model

A **session** is a per-peer interaction attempt with a finite lifecycle:
- starts at `CONNECTING`
- includes **probe** (agent-only, limited rounds)
- optionally escalates to **human entry**
- ends by timeout/close/reject/failure

A **friendship** is created only after **mutual human entry**.

---

## 4) State machine (normative)

State is tracked per `(local_agent, peer_actor_id)` pair.

### States
- `DISCONNECTED`
- `CONNECTING`
- `PROBING`
- `PROBE_COMPLETE`
- `AWAIT_LOCAL_ENTRY`
- `AWAIT_REMOTE_ENTRY`
- `MUTUAL_ENTRY_CONFIRMED`
- `FRIENDSHIP_ESTABLISHED`
- `CLOSED`
- `FAILED`

### Transitions
1. `DISCONNECTED` → `CONNECTING`
   - trigger: user initiates discovery/connection to a candidate

2. `CONNECTING` → `PROBING`
   - trigger: secure channel established and peer authenticated (key agreement done)

3. `PROBING` → `PROBE_COMPLETE`
   - trigger: probe rounds exhausted OR both sides send `probe.done`

4. `PROBE_COMPLETE` → `AWAIT_LOCAL_ENTRY`
   - trigger: local UI shows probe summary; local human has not entered

5. `AWAIT_LOCAL_ENTRY` ↔ `AWAIT_REMOTE_ENTRY`
   - trigger: local human enters (send `human.entry`) or remote human enters (receive `human.entry`)
   - system keeps track of two booleans: `local_entered`, `remote_entered`

6. `(AWAIT_*)` → `MUTUAL_ENTRY_CONFIRMED`
   - condition: `local_entered == true AND remote_entered == true`

7. `MUTUAL_ENTRY_CONFIRMED` → `FRIENDSHIP_ESTABLISHED`
   - trigger: local node persists friend record, then emits `friendship.establish`

8. Any state → `CLOSED`
   - trigger: close conditions (timeouts / explicit close / policy)

9. Any state → `FAILED`
   - trigger: transport failure, decrypt failure, auth failure, protocol violation

### Invariants
- Friendship MUST NOT be created unless both humans entered.
- Probe MUST NOT exceed its maximum rounds.
- When closed/failed, node MUST stop sending session messages (except an optional final `session.close`).

## 5) Message envelope (normative)

All messages use a signed envelope. Payload is encrypted by default.

### Envelope shape
```json
{
  "v": "0.4.3",
  "type": "probe.hello",
  "msg_id": "uuid",
  "session_id": "uuid",
  "ts": "2026-03-13T00:00:00Z",

  "from": {
    "actor_id": "h:<hash>",
    "agent_label": "optional (non-identifying display label)",
    "key_fpr": "fingerprint"
  },
  "to": {
    "actor_id": "h:<hash>",
    "key_fpr": "fingerprint"
  },

  "crypto": {
    "enc": "aead",
    "kdf": "...",
    "nonce": "base64",
    "aad": "base64(optional)"
  },

  "body": {
    "ciphertext": "base64",
    "content_type": "application/json"
  },

  "sig": "base64(signature_over_canonical_envelope_without_sig)"
}
```

### Envelope schema (normative)

#### Plaintext fields (allowed outside ciphertext)
The following may appear in plaintext:
- routing / session: `v`, `type`, `msg_id`, `session_id`, `ts`
- minimal sender/recipient identifiers: `from.actor_id`, `to.actor_id`, plus key fingerprints (`key_fpr`)
- cryptographic parameters needed to decrypt: `crypto.*` (e.g. nonce)
- ciphertext container: `body.ciphertext`, `body.content_type`
- integrity/auth: `sig`

Everything else SHOULD live inside ciphertext.

#### Ciphertext body
- The decrypted body is JSON specific to each `type`.
- Implementations MUST treat any decrypted text as untrusted input.

#### Signature
- Use **RFC 8785 JCS** to canonicalize the envelope for signing.
- Signature input is the canonical JSON of the envelope with `sig` omitted:
  - `sig = Sign( UTF8(JCS(envelope_without_sig)) )`

### Envelope rules
- `from.actor_id` and `to.actor_id` MUST be `actor_id` values (no raw handles).
- `agent_label` MUST be non-identifying; MUST NOT contain raw handle.
- `ts` MUST be present.
- `msg_id` MUST be unique (dedupe).
- `session_id` binds all probe/entry/close messages.
- `body.ciphertext` MUST be present for all non-error messages.
- Implementations SHOULD minimize plaintext metadata (only routing essentials).

## 6) Message types (normative list)

All message bodies below are the **decrypted JSON** inside `body.ciphertext`.

### Probe phase
- `probe.hello`
- `probe.question`
- `probe.answer`
- `probe.summary`
- `probe.done`

### Human entry + friendship
- `human.entry`
- `human.exit` (optional)
- `friendship.establish`

### Session control
- `session.close`
- `error`

## 7) Probe (finite rounds)

### Round limit
- Probe MUST be finite.
- Define `MAX_PROBE_ROUNDS` (default: **5 rounds** total question/answer turns per side; tune later).
- When the limit is reached, node MUST send `probe.done` and transition to `PROBE_COMPLETE`.

### Allowed content boundary (per round)
Probe is **minimal pre-chat** and MUST NOT:
- exchange contact methods (phone/email/telegram/wechat/etc)
- exchange raw handles
- exchange friend lists
- exchange files
- request sensitive personal info (address, employer, precise location, IDs)

Probe MAY exchange (bounded):
- intent summary (why connect) — short text
- language preferences
- high-level conversation mode preference (sync/async)
- safety preferences (e.g. “no sensitive topics”)
- capability flags (protocol versions, supported transports)

### Suggested per-round schema constraints (normative-ish)
- `probe.question.q`: max 300 chars
- `probe.answer.a`: max 500 chars
- No URLs by default (optional later)

### Human observability
- Local UI MUST display:
  - each probe message (redacted if needed)
  - a generated `probe.summary`
  - current remaining rounds + timeout

### `probe_transcript_hash` (normative)
Purpose: bind `human.entry` to the exact probe transcript **without recording or hashing any plaintext probe text**.

#### Canonical JSON
- Use **RFC 8785 JSON Canonicalization Scheme (JCS)** for canonical JSON serialization.

#### Message ordering
- Consider the set `P` of all envelopes in this `session_id` whose `type` starts with `probe.`.
- Order messages by:
  1) `ts` ascending (string compare on RFC3339/ISO timestamps)
  2) then `msg_id` ascending (lexicographic)

#### Per-message hash (no text)
For each probe envelope `e` in ordered `P`, compute:
- `e_core = {
    v, type, msg_id, session_id, ts,
    from: { actor_id, key_fpr },
    to:   { actor_id, key_fpr },
    crypto: { enc, kdf, nonce },
    body: { ciphertext }
  }`
- `msg_hash = SHA-256( UTF8( JCS(e_core) ) )`

Notes:
- This uses ciphertext only; it does not depend on decrypted JSON or any text fields.
- `sig` is intentionally excluded (so `probe_transcript_hash` is stable even if signature encoding changes).

#### Transcript hash
- `transcript_core = { session_id, v: "0.4.3", probe: [ { i, ts, type, msg_hash, ciphertext_len } ... ] }`
  - where `ciphertext_len` is the **byte length of the decoded ciphertext** (i.e. base64-decode then count bytes).
- `probe_transcript_hash = SHA-256( UTF8( JCS(transcript_core) ) )`

#### Binding requirement
- `human.entry.bind.probe_transcript_hash` MUST equal the locally computed `probe_transcript_hash` at time of entry.

## 8) Human entry rules (normative)

### Definition
**Human entry** means the local human explicitly chooses to join the conversation.

### Rules
- Friendship Node MUST NOT create friendship before both sides have human entry.
- Human entry MUST be explicit UI action.
- Human entry MUST be communicated to peer via `human.entry`.

### `human.entry` body
```json
{
  "entered": true,
  "entry_id": "uuid",
  "note": "optional short note (non-identifying)",
  "bind": {
    "session_id": "uuid",
    "probe_transcript_hash": "hash of probe transcript/summary"
  }
}
```

## 9) Timeout / close conditions (normative)

### Timeouts
- `PROBE_TIMEOUT`: maximum wall time allowed in `PROBING` (default: **8 minutes**)
- `ENTRY_TIMEOUT`: maximum wall time to wait for human entry after probe (default: **20 minutes**)
- If timeouts hit, node MUST close the session.

### Automatic close (required)
- If long time passes with **no human entry**, session MUST end automatically (`session.close` with reason `NO_HUMAN_ENTRY_TIMEOUT`).

### `session.close` body
```json
{
  "reason": "NO_HUMAN_ENTRY_TIMEOUT | PROBE_TIMEOUT | USER_CLOSE | POLICY_REJECT | TRANSPORT_FAIL | PROTOCOL_VIOLATION",
  "detail": "optional",
  "final": true
}
```

## 10) Friendship establishment (normative)

### Conditions (must all be true)
- `local_entered == true`
- `remote_entered == true`
- session not closed/failed

### Action
- Local node MUST write a Friend record to local storage.
- Local node SHOULD emit `friendship.establish` to peer.

### `friendship.establish` body
```json
{
  "friend_id": "uuid",
  "peer_actor_id": "h:<hash>",
  "created_at": "...",
  "session_id": "uuid",
  "notes": "optional"
}
```

### Graph propagation ban
- Friendship establish MUST NOT include friend lists.
- Node MUST NOT request peer friend lists.

## 11) Local storage — minimal schema (normative minimum)

Storage is strictly local.

### `friends`
- `friend_id` (pk)
- `peer_actor_id` (unique)
- `peer_key_fpr` (last seen)
- `created_at`
- `last_seen_at`
- `status` (FRIEND | BLOCKED | REMOVED)
- `notes` (optional)

### `sessions`
- `session_id` (pk)
- `peer_actor_id`
- `state`
- `created_at`
- `updated_at`
- `probe_rounds_used`
- `probe_transcript_hash` (hash; binds human entry to probe transcript)
- `local_entered` (bool)
- `remote_entered` (bool)
- `closed_reason` (nullable)

### `audit_logs` (minimal)
- `log_id` (pk)
- `session_id`
- `ts`
- `event_type` (MSG_SENT | MSG_RECV | STATE_CHANGE | HUMAN_ACTION)
- `event_hash` (hash of safe/redacted payload)
- `preview_safe` (optional; MUST be metadata only: type/round/length/hash; MUST NOT include any text snippets)

> Optional later: store encrypted full transcript. MVP may store only hashes + redacted preview.

## 12) Raw handle prohibition (hard rule)

### MUST NOT
- Messages MUST NOT contain raw handle in any field (including labels, notes, errors).
- Nodes MUST NOT log raw handle in plaintext logs.

### SHOULD
- Enforce a validation/lint step on outbound messages and UI text:
  - reject if it matches patterns of emails/phone/handles
  - reject if it includes user-provided identifiers

---

## 13) Discovery (MVP: minimal candidate discovery)

This spec only defines the session semantics after a candidate is chosen.

MVP discovery is intentionally minimal:
- candidate is provided out-of-band (QR / invite string / manual entry)
- no friend-of-friend
- no sharing friend lists
- no recommendation

The discovery mechanism MUST NOT leak raw handle.
